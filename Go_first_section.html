Topic 01

Why Go? (Day 1 fundamentals)

Go (Golang) is built around five practical strengths: fast compilation, fast startup, raw performance, built-in concurrency, and a simple static type system. Here's the corrected, complete picture:

  • Fast compilation: Go compiles quickly compared to many languages, which matters a lot inside CI/CD pipelines — faster builds mean shorter feedback loops, less compute spent waiting, and quicker shipping.
  • Fast startup: Go compiles straight to a single, static, native binary. There's no virtual machine or interpreter to boot (unlike the JVM or a Python interpreter), so right after deployment the program is ready to accept requests almost instantly. This is a big deal for containers and serverless cold starts.
  • Performance: Go source compiles directly to machine code, not to bytecode that a VM interprets — so it runs close to C-level speed.
  • Concurrency: Go's concurrency model is built around lightweight goroutines managed by the Go runtime's own scheduler, plus channels for passing data between them safely. This is Go's implementation of the CSP (Communicating Sequential Processes) model.
  • Static typing: types are checked at compile time, catching mismatches before the program ever runs — while := still gives you type inference so it doesn't feel as verbose as some other statically typed languages.
  • Entry point: every executable Go program needs exactly one package main with a func main() — that combination is the entry point. Everything else lives in importable packages, including built-ins like fmt for formatted I/O.
Good to know

Your notes said goroutines give Go "the best concurrency model" — that's an opinion, not a fact, and it leaves out half the story: goroutines on their own just run things concurrently. The other half is channels, which let goroutines communicate and synchronize without manually managing locks. Interviewers almost always pair these two questions, so know both.

hello.go
package main

import "fmt"

func main() {
    fmt.Println("Hello, Go!")
}
Good to know

go run main.go compiles and immediately executes the program — no binary is left behind. go build main.go only compiles, producing a standalone executable (e.g. main) that you run separately with ./main. That binary needs no Go installation to run — it's fully self-contained, which is exactly why Go is so popular for Docker images.

Interview Questions — Why Go

QWhy is Go popular for backend and cloud-native services?⌄
The combination matters more than any single trait: fast compile times speed up CI/CD, the compiled binary starts instantly with no runtime dependency, performance is close to C, and goroutines/channels make concurrent, networked code simple to write and cheap to run at scale.
QWhat's the difference between go run and go build?⌄
go run compiles to a temporary location and executes immediately — nothing persists. go build compiles and writes a persistent, directly executable binary to disk, but does not run it.
QWhat is the entry point of a Go program?⌄
A package main containing a func main(). The Go toolchain only treats a package as a buildable executable if it's named main and defines that function.
QWhy are goroutines described as "lightweight" compared to OS threads?⌄
A goroutine starts with a tiny stack (a couple of KB) that grows and shrinks as needed, and the Go runtime multiplexes many goroutines onto a much smaller number of OS threads (M:N scheduling). OS threads, by contrast, typically reserve megabytes of stack and are scheduled by the operating system, so spawning thousands of them is far more expensive than spawning thousands of goroutines.
QHow do goroutines typically communicate with each other?⌄
Through channels — typed conduits created with make(chan T) that one goroutine sends values into and another receives from. Go's philosophy is summed up as "don't communicate by sharing memory; share memory by communicating," which is why channels are preferred over manually guarding shared variables with locks.
QWhat does static typing buy you, and how does Go avoid the usual verbosity that comes with it?⌄
Static typing catches type mismatches at compile time instead of at runtime, which improves reliability and enables better tooling. Go offsets the usual verbosity with type inference — name := "value" lets the compiler work out the type, so you rarely have to write the type explicitly.
Topic 02

Variables & Constants

Go gives you three equivalent ways to declare a variable, and they exist for slightly different situations:

variables.go
var name string = "mujtaba"   // explicit type
var name = "mujtaba"          // type inferred from the value
name := "mujtaba"             // short form — only valid inside a function

// useful when you want to assign the value later
var age int
age = 25
  • var name string = "mujtaba" — spells out the type explicitly. Handy when you want to declare first and assign later, e.g. var name string then name = "ali".
  • var name = "mujtaba" — same idea, but the type is inferred from the value.
  • name := "mujtaba" — the short declaration. It infers the type too, but it can only be used inside a function body — at package level (outside any function) Go requires var or const.

If you declare a variable without assigning it a value, Go gives it the zero value for its type instead of leaving it undefined: 0 for numeric types, "" for strings, false for booleans, and nil for pointers, slices, maps, channels, and functions.

Your note about unused variables is correct and worth remembering precisely: Go's compiler refuses to build if a local variable is declared and never used — this is a compile error, not a warning. The same rule does not apply to package-level (global) variables.

Go also has multiple sized integer types (int8, int16, int32, int64 and unsigned equivalents). By default you reach for plain int, whose size (32 or 64-bit) is chosen by the compiler for the target platform — you only pick a specific width when you have a concrete reason, like matching a binary protocol or saving memory at scale.

constants.go
const port = 5000

const (
    Port = 5000
    Host = "localhost"
)

A const value is fixed at compile time and can never be reassigned — attempting to do so is a compile error.

Good to know

Inside a const block, Go has a special identifier called iota that auto-increments with each line — it's the idiomatic way to build enums, and it's a near-guaranteed interview question whenever const comes up:

type Status int

const (
    Pending Status = iota // 0
    Active                // 1
    Closed                // 2
)

Interview Questions — Variables & Constants

QWhat are the three ways to declare a variable in Go, and when would you reach for each?⌄
var name type = value when you want to be explicit or declare before assigning; var name = value when you want inference but are outside a function; name := value as the everyday short form, restricted to function bodies.
QWhy can't you use := at package level?⌄
The short declaration form is a statement, and Go's grammar only allows statements inside function bodies. Package-level declarations must use the var or const keyword instead.
QWhat is a "zero value" in Go? Give a few examples.⌄
The default value Go automatically assigns to a variable that's declared but not initialized. Examples: 0 for int/float, "" for string, false for bool, and nil for pointers, slices, maps, channels, interfaces, and functions.
QDoes Go allow an unused local variable to compile?⌄
No — it's a compile-time error, by design, to keep code free of dead variables. Unused package-level variables and unused imports follow similar strictness (unused imports are also a compile error).
QWhat's the difference between const and a regular variable?⌄
A const value is fixed at compile time, can never be reassigned, and must be a value the compiler can determine without running the program (numbers, strings, booleans, or expressions of those) — it can't be the result of a function call at runtime.
QWhat does iota do?⌄
Inside a const (...) block, iota starts at 0 and increments by one for every line, making it the standard way to define a sequence of related constants (enums) without writing out each number.
Topic 03

Loops

Go has exactly one looping keyword, for — there is no separate while or do-while. You shape for into whichever style you need.

loops.go
// "while" style
i := 1
for i <= 3 {
    fmt.Println(i)
    i++
}

// infinite loop — exit with break
for {
    // ...
    break
}

// classic three-clause for
for i := 0; i < 3; i++ {
    fmt.Println(i)
}

// range over an integer (Go 1.22+)
for i := range 3 {
    fmt.Println(i) // 0, 1, 2
}

break and continue work as in most C-family languages: break exits the loop entirely, continue skips to the next iteration.

Good to know

Plain continue/break only affect the innermost loop. To control an outer loop from inside a nested one, label the outer loop:

outer:
for i := 0; i < 3; i++ {
    for j := 0; j < 3; j++ {
        if j == 1 {
            continue outer
        }
        fmt.Println(i, j)
    }
}

Interview Questions — Loops

QHow many looping constructs does Go provide?⌄
Just one — for. It's flexible enough to act as a while-loop, an infinite loop, a classic counted loop, or a range-based loop depending on what you put around it.
QHow do you write an infinite loop in Go?⌄
for { ... } with no condition at all — you exit it explicitly with a break or a return.
QWhat's the difference between break and continue?⌄
break stops the loop completely and moves execution to the line after it. continue stops the current iteration only and jumps straight to the loop's next iteration check.
QHow do you break or continue an outer loop from inside a nested loop?⌄
Label the outer loop (e.g. outer: on the line above it) and use break outer or continue outer from inside the nested loop.
QWhat is "range over integers" and when was it added?⌄
Since Go 1.22, for i := range n iterates i from 0 to n-1 — a concise alternative to a classic three-clause counting loop when you don't need a custom step.
Topic 04

If / Else

Go's if/else reads like most C-family languages, with two syntax differences worth remembering for an interview: parentheses around the condition are not used, and curly braces are never optional — even for a single-line body.

if-else.go
if age > 18 {
    fmt.Println("person is an adult")
} else if age == 18 {
    fmt.Println("just became an adult")
} else {
    fmt.Println("not an adult yet")
}

// a variable declared in the if's init statement
// is scoped only to the if/else block
if age := 15; age > 18 {
    fmt.Println("adult")
} else {
    fmt.Println("not an adult:", age)
}

Interview Questions — If / Else

QAre parentheses required around an if-condition in Go?⌄
No — and idiomatic Go omits them entirely. Writing if (age > 18) compiles, but gofmt/reviewers will flag the unnecessary parentheses.
QCan you omit the curly braces for a single-statement if-body, like in C?⌄
No. Go requires braces on every if/else body regardless of length, and the opening brace must be on the same line as the condition — Go's grammar actually depends on this for unambiguous parsing.
QWhat's the point of writing a short statement before the condition, like if age := 15; age > 18 {}?⌄
It declares a variable scoped only to that if/else statement — useful for values you only need within the branch, like the result of a function call you're about to check. The variable isn't visible outside the if/else block.
Topic 05

Logical Operators

Go has the three logical operators you'd expect — AND (&&), OR (||), and NOT (!) — and, deliberately, no ternary operator.

logical.go
isAdult := age >= 18 && hasID
canEnter := isAdult || hasGuardian
isMinor := !isAdult

Go's designers left out the ternary (condition ? a : b) on purpose, arguing it's often abused to cram logic into unreadable one-liners. The idiomatic replacement is a plain if/else, or a small helper function if you need the expression form repeatedly.

Interview Questions — Logical Operators

QDoes Go have a ternary operator?⌄
No. Go intentionally omits it for readability; you use a full if/else statement, or write a tiny generic helper function if you really want a conditional expression.
QDo && and || short-circuit in Go?⌄
Yes. && stops evaluating as soon as one operand is false, and || stops as soon as one operand is true — the same short-circuit behavior as in C, Java, and JavaScript.
QWhat are Go's three logical operators?⌄
&& (AND), || (OR), and ! (NOT) — all of which only operate on boolean expressions, since Go does not auto-convert other types to bool.
Topic 06

Switch Statements

The shape of a Go switch matches other languages, including grouping several values into one case:

switch.go
switch i {
case 1:
    fmt.Println("one")
case 2:
    fmt.Println("two")
default:
    fmt.Println("something else")
}

// multiple values in a single case
switch i {
case 1, 2:
    fmt.Println("one or two")
default:
    fmt.Println("other")
}
Good to know

This is the single biggest switch-related gotcha in interviews, and it wasn't in your notes at all: unlike C, Java, or JavaScript, a Go case does not fall through to the next one automatically — each case implicitly stops after its block runs. If you genuinely want fallthrough behavior, you opt in explicitly with the fallthrough keyword:

switch i {
case 1:
    fmt.Println("one")
    fallthrough
case 2:
    fmt.Println("also runs for 1, because of fallthrough")
}

A switch can also drop the value entirely, which turns it into a cleaner-looking if/else chain:

tagless switch
switch {
case age < 13:
    fmt.Println("child")
case age < 20:
    fmt.Println("teenager")
default:
    fmt.Println("adult")
}

And a type switch branches on the dynamic type stored inside an interface value:

type-switch.go
var i interface{} = 10

switch v := i.(type) {
case int:
    fmt.Println("int:", v)
case bool:
    fmt.Println("bool:", v)
default:
    fmt.Println("unknown type")
}

This only makes sense when the switched-on value has an interface type (like interface{}, also written any) — concrete types like a plain int have nothing to switch on, since their type is already fixed at compile time.

Interview Questions — Switch Statements

QDoes a Go switch fall through to the next case automatically, like C?⌄
No — it's the opposite of C. Each matched case runs its block and then stops; you only get fallthrough behavior by explicitly writing fallthrough as the last statement in a case.
QHow do you match multiple values with one case?⌄
List them comma-separated on the same case line: case 1, 2: matches either value.
QWhat does a switch with no expression after the keyword do?⌄
It becomes a "tagless" switch where each case holds its own boolean condition, evaluated top to bottom — functionally identical to an if/else-if chain but often considered more readable.
QWhat is a type switch, and when do you reach for one?⌄
A switch of the form switch v := i.(type) that branches based on the concrete type currently stored in an interface value i. It's used whenever a function accepts an interface{}/any parameter and needs different behavior depending on what was actually passed in.
QCan a single case in a type switch match more than one type?⌄
Yes, e.g. case int, int64: — but inside that combined case, the variable keeps the interface type rather than being narrowed to one specific type, since the compiler can't know which of the two it is.
Topic 07

Arrays

arrays.go
var nums [4]int
nums[0] = 10
fmt.Println(len(nums)) // 4

nums2 := [3]int{1, 2, 3}

// 2D array
grid := [2][2]int{{1, 2}, {3, 4}}
Good to know

This is the detail most beginner notes skip, and it's a favorite interview question: in Go, an array's length is part of its type. [4]int and [5]int are different, incompatible types — you can't assign one to the other, and a function expecting [4]int will not accept a [5]int. Arrays are also value types: assigning one to another variable, or passing one into a function, copies every element.

a := [3]int{1, 2, 3}
b := a       // full copy
b[0] = 99
fmt.Println(a[0]) // still 1 — "a" is untouched

Because the size is fixed and baked into the type, arrays are used far less often in everyday Go than slices — reach for an array only when you know the exact, unchanging size up front (a fixed-size buffer, a checksum, coordinates).

Interview Questions — Arrays

QIs an array's size part of its type in Go? What does that imply?⌄
Yes. [4]int and [5]int are distinct types, so you can't assign between them, and a function parameter typed [4]int will reject a [5]int argument.
QAre arrays passed by value or by reference when you pass them to a function?⌄
By value — the entire array is copied. Mutating the parameter inside the function does not affect the caller's original array unless you explicitly pass a pointer to it.
QWhen would you choose an array over a slice?⌄
When the size is fixed and known ahead of time and won't change — e.g. a 3-element RGB value or a fixed-size hash. For anything that might grow or whose size isn't known upfront, a slice is the idiomatic choice.
QHow do you get the length of an array?⌄
len(arr) — the same built-in function used for slices, strings, and maps.
Topic 08

Slices

A slice is a dynamically-sized view over an underlying array — it's what you reach for almost every time you'd use a list/array in another language.

slices.go
var nums []int            // nil slice — len 0, cap 0
fmt.Println(nums == nil)  // true

nums = make([]int, 2)     // length 2, both elements zero-valued
nums = make([]int, 2, 5)  // length 2, capacity 5
fmt.Println(cap(nums))    // 5

nums = append(nums, 3)    // grows automatically once capacity is exceeded

literal := []int{1, 2, 3}

dest := make([]int, 3)
copy(dest, literal)

// slicing operator — low is inclusive, high is exclusive
sub := literal[0:2] // []int{1, 2}
Correction

Your notes wrote a 2D slice as var nums=[][]{{1,3,4},{1,2,3}} — that's missing the element type and won't compile. A slice type always needs the element type after the brackets:

matrix := [][]int{{1, 3, 4}, {1, 2, 3}}

Also, "checking if a slice is empty by comparing to nil" is only half the story. A nil slice (var nums []int) and an empty-but-non-nil slice (nums := []int{}) both behave like empty collections — len() is 0 for both — but only the first one equals nil. If you just want to know "is this empty", checking len(nums) == 0 is the safer, more idiomatic test.

Good to know

Two slice details that come up constantly in interviews:

Growth: when append would exceed the current capacity, Go allocates a new, larger backing array and copies everything over — which is exactly why you must always capture the result, nums = append(nums, x), rather than ignoring it: the returned slice may point at completely new memory.

Aliasing: a sub-slice created with s[low:high] shares the same backing array as the original. Mutating an element through the sub-slice can silently change the original slice too — a classic Go gotcha worth knowing cold.

Interview Questions — Slices

QWhat's the difference between an array and a slice in Go?⌄
An array has a fixed length that's part of its type and is copied by value. A slice is a small header (pointer, length, capacity) describing a view into an underlying array, can grow dynamically with append, and is copied by reference to that header — copying a slice doesn't copy the underlying data.
QWhat three things make up a slice internally?⌄
A pointer to the underlying array, a length (how many elements are currently accessible), and a capacity (how many elements the underlying array can hold before a new allocation is needed).
QWhat happens when you append past a slice's capacity?⌄
Go allocates a new, larger underlying array, copies the existing elements into it, and returns a slice pointing at that new array. The old array is left untouched, which is why you must reassign the result of append.
QWhy is it wrong to write append(nums, x) without reassigning it?⌄
Because append may or may not allocate a new backing array — if it does, the original nums variable still points at the old array and never sees the appended value. You always need nums = append(nums, x).
QWhat's the difference between len() and cap() on a slice?⌄
len() is the number of elements currently in the slice. cap() is how many elements the underlying array could hold starting from the slice's first element, before append would need to reallocate.
QCan two slices share the same underlying array? What problem can that cause?⌄
Yes — slicing an existing slice (sub := s[1:3]) creates a new slice header that still points at the same backing array. Writing to an element through either slice can unexpectedly affect the other, since they're really looking at the same memory.
Topic 09

Maps

maps.go
m := make(map[string]int)
m["a"] = 1

// reading a missing key gives the zero value, not an error
fmt.Println(m["missing"]) // 0

// "comma-ok" idiom — the real way to check existence
val, ok := m["missing"]
fmt.Println(val, ok) // 0 false

delete(m, "a")
fmt.Println(len(m))
clear(m)

// map literal, no make() needed
scores := map[string]int{
    "alice": 90,
    "bob":   85,
}
Good to know

Your notes correctly said a missing key returns the zero value rather than an error — but didn't mention how you tell "the key holds the zero value" apart from "the key doesn't exist at all". That's the comma-ok idiom: val, ok := m[key] gives you a second boolean, ok, which is false only when the key is absent.

Two more essentials that weren't in the notes: delete(m, key) removes a single entry, and iteration order over a map (with range) is intentionally randomized by the Go runtime — never rely on it being consistent between runs.

Maps are reference types, like slices — passing one into a function doesn't copy its contents, so mutations inside the function are visible to the caller. Map keys must also be a comparable type (supports ==), which rules out slices, maps, and functions as keys.

Interview Questions — Maps

QHow do you tell whether a key exists in a map versus just getting its zero value?⌄
Use the comma-ok idiom: val, ok := m[key]. ok is true if the key exists (even if its value happens to be the zero value) and false if it doesn't.
QHow do you remove a key from a map?⌄
The built-in delete(m, key) — it's a no-op (no panic) if the key doesn't exist.
QIs the iteration order of a Go map guaranteed?⌄
No — it's deliberately randomized by the runtime so that code never accidentally comes to depend on a particular order. If you need a predictable order, collect and sort the keys yourself.
QWhat types are valid as map keys?⌄
Any comparable type — one that supports == — such as strings, numbers, booleans, and structs made entirely of comparable fields. Slices, maps, and functions are not comparable and can't be used as keys.
QAre maps passed by value or reference to a function?⌄
A map variable is really a reference to an underlying hash table. Passing it into a function copies the reference, not the data, so changes made inside the function (adding/removing/updating keys) are visible to the caller.
Topic 10

The Range Keyword

range is Go's universal iteration tool — it works over arrays, slices, maps, strings, channels, and (since Go 1.22) plain integers.

range.go
nums := []int{10, 20, 30}

for _, num := range nums {
    fmt.Println(num) // _ ignores the index
}

for i, num := range nums {
    fmt.Println(i, num)
}

m := map[string]string{"a": "apple", "b": "banana"}
for k, v := range m {
    fmt.Println(k, v) // order is not guaranteed
}

// ranging over a string gives a byte index and a rune
for i, c := range "golang" {
    fmt.Println(i, c)
}
Good to know

Your note that "i is the byte index and c is the unicode value" is correct and is exactly the nuance interviewers test: Go strings are UTF-8 byte sequences, and most non-English characters take more than one byte. So when ranging over a string with such characters, the index can jump by more than 1 between iterations — it tracks bytes, not "characters":

for i, c := range "héllo" {
    fmt.Println(i, string(c))
}
// 0 h
// 1 é   (é takes 2 bytes, so the next index jumps to 3)
// 3 l
// 4 l
// 5 o

Each value c has type rune — an alias for int32 representing a single Unicode code point — which is why ranging over a string is the safe way to walk character-by-character without manually juggling UTF-8 byte boundaries.

Interview Questions — Range

QWhat types can you use range on?⌄
Arrays, slices, strings, maps, channels — and, since Go 1.22, plain integers (for i := range n).
QWhen ranging over a string, what do the two returned values actually represent?⌄
The first value is the byte index where the current rune starts in the underlying UTF-8 encoding; the second is the rune (Unicode code point) itself — not a plain ASCII byte.
QWhy might the index skip by more than 1 between iterations when ranging over a string?⌄
Because Go strings are UTF-8 encoded, and any character outside the basic ASCII range can take 2, 3, or 4 bytes. The index tracks byte position, so it jumps ahead by however many bytes the previous rune occupied.
QWhat is a rune in Go?⌄
An alias for int32 that represents a single Unicode code point, as opposed to a byte (alias for uint8), which represents a single raw byte of a UTF-8-encoded string.
QHow do you range over just the keys of a map, ignoring the values?⌄
for k := range m { ... } — when ranging a map you can omit the value entirely and just take the key.
Topic 11

Functions

Go puts the return type after the parameter list, and parameters of the same type can share one type annotation:

functions.go
func add(a int, b int) int {
    return a + b
}

// shorthand when parameters share a type
func add(a, b int) int {
    return a + b
}

If no return type is written, the function returns nothing at all — Go has no void keyword, you simply omit the return type. Go functions can also return more than one value, which your notes captured correctly:

multiple-returns.go
func get() (string, int) {
    return "go", 1
}
Good to know

This is the single most important pattern built on multiple return values, and it wasn't in the notes at all: Go has no exceptions, so the idiomatic way to signal failure is to return an error as the last return value and check it immediately. This shows up in nearly every Go codebase and is almost guaranteed to come up in an interview.

func divide(a, b int) (int, error) {
    if b == 0 {
        return 0, errors.New("division by zero")
    }
    return a / b, nil
}

result, err := divide(10, 0)
if err != nil {
    fmt.Println("error:", err)
}

You can also name return values in the signature and use a bare return — useful for documenting intent and avoiding repetition in functions with several exit paths:

func divideNamed(a, b int) (result int, err error) {
    if b == 0 {
        err = errors.New("division by zero")
        return
    }
    result = a / b
    return
}

Functions are first-class values in Go: you can pass them as arguments and return them from other functions.

functions-as-values.go
func process(fn func(int) int) {
    fmt.Println(fn(1))
}

double := func(a int) int {
    return a * 2
}
process(double)

// returning a function
func makeAdder(base int) func(int) int {
    return func(a int) int {
        return a + base
    }
}

addFive := makeAdder(5)
fmt.Println(addFive(3)) // 8

A variadic function accepts a variable number of arguments of the same type, received internally as a slice:

variadic.go
func sum(nums ...int) int {
    total := 0
    for _, n := range nums {
        total += n
    }
    return total
}

sum(1, 2, 3)

// "spreading" an existing slice into a variadic call
values := []int{4, 5, 6}
sum(values...)

Interview Questions — Functions

QHow does Go signal that a function "returns nothing"?⌄
By simply omitting the return type from the signature — Go has no void keyword. func log(msg string) { ... } returns nothing.
QWhy is the multiple-return-value pattern central to how Go handles errors?⌄
Go has no exception mechanism. Instead, any operation that can fail returns an error as its final return value (nil on success), and the caller is expected to check it immediately with if err != nil rather than relying on a catch block elsewhere.
QWhat are named return values, and what does a "bare return" do?⌄
Named return values give the return variables names directly in the function signature (e.g. (result int, err error)); they start at their zero values automatically. A bare return with no arguments then returns whatever those named variables currently hold.
QHow do you pass a function as an argument to another function?⌄
Functions are first-class values in Go, so you declare the parameter with a function type, e.g. func process(fn func(int) int), and pass any matching function or function literal as the argument.
QWhat is a variadic function, and how do you call it with values already in a slice?⌄
A function declared with ...Type on its last parameter, which lets the caller pass any number of arguments of that type (received inside as a slice). If you already have a slice, you "spread" it into the call with fn(mySlice...) instead of passing each element manually.
QWhy would a function return another function? Give a real use case.⌄
It's how Go builds configurable behavior and middleware — a function like makeAdder(base) returns a specialized function that remembers base. The same pattern underlies HTTP middleware, where a function wraps a handler and returns a new handler with extra behavior attached.
Topic 12

Closures

Correction

Your notes defined a closure as "those which don't receive anything but return a function" — that's actually just one common pattern built using a closure, not the definition. The real definition: a closure is a function value that references variables declared outside its own body — it "closes over" them, keeping access to (and the ability to mutate) those variables even after the outer function has returned. A closure can take parameters, and doesn't have to return another function at all.

The counter example from your notes is a textbook closure — it's just one pattern, not the whole definition:

closures.go
func counter() func() int {
    count := 0

    return func() int {
        count++
        return count
    }
}

c1 := counter()
fmt.Println(c1()) // 1
fmt.Println(c1()) // 2

c2 := counter() // a brand-new, independent "count"
fmt.Println(c2()) // 1

The inner anonymous function captures count by reference, not by copy — every call to c1() shares and mutates the same count that lives inside that particular closure. Because counter() creates a fresh count on every call, c1 and c2 are completely independent counters.

Good to know

A classic Go interview trap is the loop-variable-capture bug. Before Go 1.22, every iteration of a range loop reused the same loop variable, so closures created inside the loop (like goroutines) could all end up capturing the final value instead of "their own":

for _, v := range []int{1, 2, 3} {
    go func() {
        fmt.Println(v)
    }()
}
// Pre-1.22: commonly printed 3, 3, 3 (in some order)
// Go 1.22+: each iteration gets its own "v", so this now
// reliably prints 1, 2, 3 (in some order)

Knowing this changed in Go 1.22 is a good way to show you're current — it's one of the most-cited gotchas in older Go code.

Interview Questions — Closures

QWhat is a closure, precisely?⌄
A function value that captures and references variables from the scope it was defined in, retaining access to (and the ability to modify) those variables even after the enclosing function has finished executing.
QIn the counter() example, why are c1 and c2 independent of each other?⌄
Each call to counter() creates a brand-new count variable and a brand-new inner function closing over that specific variable. c1 and c2 close over two entirely separate count variables, so incrementing one never affects the other.
QDoes a closure copy the variables it captures, or reference them?⌄
By reference. The closure shares the actual variable with the scope that created it, which is exactly why calling c1() repeatedly keeps incrementing the same underlying count instead of resetting it.
QWhat was the classic loop-variable-capture bug with closures in Go, and is it still a problem?⌄
Before Go 1.22, a range loop reused one loop variable across all iterations, so goroutines or closures created inside the loop often captured the same final value instead of the value from their own iteration. Go 1.22 changed the semantics so each iteration gets its own copy of the loop variable, which fixed the most common version of this bug going forward — but it's still important to recognize in pre-1.22 code.
QGive a practical use case for closures beyond counters.⌄
Closures are how Go implements things like HTTP middleware (a function that wraps a handler and returns a new handler with shared state, like a logger or auth check), memoization caches, and configurable callbacks where some setup state needs to persist between calls.
