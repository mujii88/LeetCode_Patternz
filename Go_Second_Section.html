Pointers

A pointer stores the memory address of another variable. In Go, all function arguments are passed by value (a copy is made). If you need a function to modify the original variable, you pass a pointer instead.

💡
Use &variable to get its memory address, and *pointer to dereference it (access or change the value it points to).
OperatorNameWhat it does
&Address-ofReturns the memory address of a variable
*TPointer typeDeclares a pointer to a value of type T
*ptrDereferenceReads or writes the value at the pointer's address

❌ Pass by Value (original not modified)

go 
package main

import "fmt"

func changeValue(a int) {
    a = 100  // only modifies the local copy
}

func main() {
    i := 1
    changeValue(i)
    fmt.Println(i)  // Output: 1  (unchanged!)
}

✅ Pass by Pointer (original is modified)

go 
package main

import "fmt"

// *int means: a is a pointer to an int
func changeValue(a *int) {
    *a = 100  // dereference and write to original
}

func main() {
    i := 1
    changeValue(&i)       // pass address of i
    fmt.Println(i)        // Output: 100  ✓

    // Pointer basics
    p := &i              // p holds the address of i
    fmt.Println(p)       // e.g. 0xc0000140a8
    fmt.Println(*p)      // 100  (dereference)
    *p = 42
    fmt.Println(i)       // 42  (i changed via pointer)
}

⚠️ Nil Pointers — Always Check!

go 
var p *int    // zero value of a pointer is nil

if p == nil {
    fmt.Println("pointer is nil — don't dereference!")
}

// Dereferencing nil causes a runtime panic:
// *p = 5  ← PANIC: invalid memory address
⚠️
Dereferencing a nil pointer causes a runtime panic in Go. Always guard pointer usage with nil checks when the pointer might not be set.
Interview Q&A — Pointers
Q1. What is the zero value of a pointer in Go? Easy ▼
The zero value of any pointer type in Go is nil. A nil pointer means the pointer doesn't point to any valid memory address. Dereferencing a nil pointer causes a runtime panic, so you should always check if ptr != nil before using it.
Q2. What is the difference between pass by value and pass by reference in Go? Easy ▼
Go is strictly pass by value — every function call receives a copy of the argument. However, you can simulate pass by reference by passing a pointer (&variable). The function then receives a copy of the pointer (the address), and can use *ptr to read/modify the original value. Slices, maps, and channels already contain internal pointers, so their contents can be modified without explicit pointer syntax.
Q3. When should you use pointers vs values in Go? Medium ▼
Use a pointer when:
  • You want a function to modify the original value
  • The struct is large (copying is expensive)
  • You want to express "optional" data (nil vs zero value)
  • All methods on a type use pointer receivers (consistency)
Use a value when:
  • The type is small (int, bool, small struct)
  • You want immutability — the function should not change the caller's data
Q4. What does the `new()` built-in do? Medium ▼
new(T) allocates memory for a value of type T, zeroes it, and returns a pointer to it (*T).

Example: p := new(int) → p is a *int pointing to a zeroed int (0). This is equivalent to var x int; p := &x. In modern Go code, new() is rarely used for structs; the composite literal pattern like &MyStruct{} is preferred instead.
Q5. Can you return a pointer to a local variable in Go? Hard ▼
Yes! This is safe in Go. Go's escape analysis detects that the local variable's address is being returned and automatically allocates it on the heap instead of the stack. The garbage collector manages its lifetime. This is a fundamental difference from C/C++ where returning a pointer to a local stack variable causes undefined behavior.

Example: func newInt() *int { x := 42; return &x } is perfectly valid in Go.

Structs

Structs are Go's answer to classes. They let you group related fields together under a named type. Go intentionally has no class keyword — structs + methods + interfaces cover everything classes do.

📖
Structs in Go are value types. Assigning one struct to another creates a full copy of all fields.

Declaring a Struct

go 
type Order struct {
    ID     int
    Name   string
    Amount float32
    Status string
}

// Anonymous/inline struct (useful for one-off values)
lang := struct {
    Name string
    Year int
}{"Go", 2009}

Creating Instances

go 
// Named fields (recommended — order-independent, self-documenting)
myOrder := Order{
    ID:     2,
    Name:   "Mujtaba",
    Amount: 99.99,
    Status: "pending",
}

// Positional (fragile — breaks if fields are reordered)
o2 := Order{3, "Ali", 50.0, "shipped"}

// Pointer to struct
op := &Order{ID: 4, Name: "Sara"}

// Access fields with dot notation
fmt.Println(myOrder.Name)    // Mujtaba
myOrder.Status = "paid"      // modify a field
fmt.Println(op.ID)           // 4 (Go auto-dereferences: op.ID = (*op).ID)

⚙️ Methods & Receivers

A method is a function with a receiver argument — it attaches the function to the struct type.

go 
// VALUE receiver — reads only, works on a copy
func (o Order) Summary() string {
    return fmt.Sprintf("Order #%d by %s — $%.2f", o.ID, o.Name, o.Amount)
}

// POINTER receiver — can modify the original struct
func (o *Order) UpdateStatus(status string) {
    o.Status = status  // no need to write (*o).Status — Go auto-dereferences
}

func main() {
    myOrder := Order{ID: 1, Name: "Mujtaba", Amount: 100}
    fmt.Println(myOrder.Summary())    // Order #1 by Mujtaba — $100.00
    myOrder.UpdateStatus("paid")
    fmt.Println(myOrder.Status)        // paid
}
💡
Rule of thumb: if any method on a type needs a pointer receiver, make all methods pointer receivers for consistency.

🔨 Constructor Functions

Go has no new keyword for structs. The convention is to write a factory function prefixed with New.

go 
// Returns a *Order — returning pointer avoids copying
// and allows methods with pointer receivers to work
func NewOrder(id int, name string, amount float32) *Order {
    if amount < 0 {
        amount = 0  // validation logic goes here
    }
    return &Order{
        ID:     id,
        Name:   name,
        Amount: amount,
        Status: "pending",  // sensible default
    }
}

func main() {
    o := NewOrder(1, "Mujtaba", 49.99)
    fmt.Println(o.Summary())
}

🧩 Struct Embedding (Composition / "Inheritance")

Go favors composition over inheritance. You embed one struct inside another to reuse its fields and methods.

go 
type Customer struct {
    Name     string
    Location string
    Email    string
}

func (c Customer) Greet() string {
    return "Hello, " + c.Name
}

// Named embedding — explicit field name
type Order struct {
    ID          int
    Amount      float32
    Customer    Customer    // named — access via order.Customer.Name
}

// Anonymous embedding — promotes fields & methods
type PremiumOrder struct {
    Order                    // anonymous — fields & methods promoted
    Discount float32
}

func main() {
    o := Order{
        ID:     1,
        Amount: 99,
        Customer: Customer{Name: "Mujtaba", Location: "Islamabad"},
    }
    fmt.Println(o.Customer.Name)    // "Mujtaba"

    po := PremiumOrder{
        Order:    o,
        Discount: 10.0,
    }
    // Promoted fields — access without specifying Order
    fmt.Println(po.Greet())      // promoted from Customer via Order
}
📌
Anonymous embedding promotes the embedded type's fields and methods to the outer struct. It's Go's mechanism for code reuse — similar to but not the same as inheritance.
Interview Q&A — Structs
Q1. What is the difference between a value receiver and a pointer receiver? Easy ▼
A value receiver (func (o Order) ...) receives a copy of the struct. Changes inside the method don't affect the caller's struct. A pointer receiver (func (o *Order) ...) receives a pointer to the original struct, so modifications persist after the method returns. Use pointer receivers when: you need to mutate state, the struct is large (avoid copying), or you need to handle nil.
Q2. Can Go structs implement interfaces? Easy ▼
Yes. In Go, interface implementation is implicit. A struct implements an interface as long as it has all the methods the interface requires — no implements keyword is needed. This is called duck typing or structural subtyping. The compiler checks at compile time whether the struct satisfies the interface.
Q3. What is the difference between struct embedding and a regular field? Medium ▼
A named field like Customer Customer requires you to access it via order.Customer.Name. An anonymous/embedded field like just Customer (no field name) promotes all of Customer's fields and methods to the outer struct — so you can write order.Name directly. Embedding also means the outer struct satisfies any interface that the embedded struct satisfies, because the promoted methods count.
Q4. Are structs comparable in Go? Medium ▼
A struct is comparable with == and != only if all its fields are comparable types (int, string, bool, pointers, etc.). Structs containing slices, maps, or functions are not comparable. Two struct values are equal if all their corresponding fields are equal. This makes structs usable as map keys when comparable.
Q5. What is the "functional options" pattern for constructors? Hard ▼
Instead of a constructor with many parameters, you pass option functions:

type Option func(*Order)
func WithStatus(s string) Option { return func(o *Order){ o.Status = s } }
o := NewOrder(1, "Ali", 50, WithStatus("paid"), WithDiscount(10))

This avoids long parameter lists, allows optional config, and keeps the API backward-compatible as you add new options without breaking existing callers.

Interfaces

An interface defines a set of method signatures. Any type that implements all those methods automatically satisfies the interface — no explicit declaration needed. This gives Go powerful polymorphism without inheritance.

💡
Why interfaces? They let you write code that works with any type satisfying a contract, without knowing the concrete type. This is the Open/Closed Principle — extend behavior by adding new types, not by modifying existing code.

Declaring an Interface

go 
// Convention: interface names often end with -er
type Paymenter interface {
    Pay(amount float32) (bool, error)
    Refund(amount float32) bool
}

🌟 Full Payment Gateway Example (Open/Closed Principle)

This is the classic example from your notes, expanded. Adding a new payment provider (PayPal, Crypto) requires zero changes to existing code.

go 
package main

import "fmt"

// ── Interface (the contract) ──────────────────────
type Paymenter interface {
    Pay(amount float32) (bool, error)
    Refund(amount float32) bool
    Name() string
}

// ── Stripe implementation ─────────────────────────
type Stripe struct{ APIKey string }

func (s Stripe) Pay(amount float32) (bool, error) {
    fmt.Printf("[Stripe] Charging $%.2f\n", amount)
    return true, nil
}
func (s Stripe) Refund(amount float32) bool {
    fmt.Printf("[Stripe] Refunding $%.2f\n", amount)
    return true
}
func (s Stripe) Name() string { return "Stripe" }

// ── PayPal implementation ─────────────────────────
type PayPal struct{ ClientID string }

func (p PayPal) Pay(amount float32) (bool, error) {
    fmt.Printf("[PayPal] Processing $%.2f\n", amount)
    return true, nil
}
func (p PayPal) Refund(amount float32) bool { return true }
func (p PayPal) Name() string            { return "PayPal" }

// ── Payment service uses the interface ────────────
type PaymentService struct {
    Gateway Paymenter    // depends on abstraction, not concrete type
}

func (ps PaymentService) Checkout(amount float32) {
    ok, err := ps.Gateway.Pay(amount)
    if err != nil || !ok {
        fmt.Println("Payment failed!")
        return
    }
    fmt.Printf("✅ Paid via %s\n", ps.Gateway.Name())
}

func main() {
    stripe := Stripe{APIKey: "sk_live_..."}
    svc := PaymentService{Gateway: stripe}
    svc.Checkout(99.99)

    // Swap to PayPal with ONE line change — no other code changes
    svc.Gateway = PayPal{ClientID: "paypal_id"}
    svc.Checkout(49.99)
}

The Empty Interface — any

go 
// interface{} (or "any" in Go 1.18+) has no methods
// so every type satisfies it automatically
func Print(v any) {
    fmt.Println(v)
}

Print(42)
Print("hello")
Print([]int{1,2,3})

Type Assertions & Type Switches

go 
func describe(p Paymenter) {
    // Type assertion — extract concrete type
    if s, ok := p.(Stripe); ok {
        fmt.Println("This is a Stripe with key:", s.APIKey)
    }

    // Type switch — cleaner for multiple types
    switch v := p.(type) {
    case Stripe:
        fmt.Println("Stripe:", v.APIKey)
    case PayPal:
        fmt.Println("PayPal:", v.ClientID)
    default:
        fmt.Println("Unknown gateway")
    }
}

Interface Composition

Interfaces can embed other interfaces, creating composite contracts.

go 
type Payer interface {
    Pay(amount float32) bool
}

type Refunder interface {
    Refund(amount float32) bool
}

// Composite interface embeds both
type PaymentGateway interface {
    Payer
    Refunder
    Name() string
}

// The Go standard library uses this extensively:
// io.ReadWriter embeds io.Reader + io.Writer

🎯 When Should You Use an Interface?

  • You have 2+ concrete types that need to be used interchangeably
  • You want to write code that works with future types not yet written
  • You want to write testable code — inject a mock instead of a real dependency
  • You want to follow Open/Closed Principle (add features by adding types, not changing existing code)
  • You're building a library and want users to be able to provide their own implementations
⚠️
Don't over-interface. Go's advice: accept interfaces, return concrete types. Only abstract when you have a real need — premature abstraction adds complexity.

🧪 Mocking for Unit Tests

One of the biggest real-world uses of interfaces — swap the real gateway for a mock in tests.

go 
// mock_payment.go (test file)
type MockPayment struct {
    ShouldFail bool
    CallCount  int
}

func (m *MockPayment) Pay(amount float32) (bool, error) {
    m.CallCount++
    if m.ShouldFail { return false, fmt.Errorf("mock failure") }
    return true, nil
}
func (m *MockPayment) Refund(amount float32) bool { return true }
func (m *MockPayment) Name() string            { return "Mock" }

func TestCheckout(t *testing.T) {
    mock := &MockPayment{}
    svc  := PaymentService{Gateway: mock}
    svc.Checkout(100)
    if mock.CallCount != 1 {
        t.Errorf("expected 1 call, got %d", mock.CallCount)
    }
}

📘 Real-world Interface: io.Writer

The standard library's most famous interface — just one method.

go 
// This is literally the io.Writer interface definition in stdlib:
type Writer interface {
    Write(p []byte) (n int, err error)
}

// os.File, bytes.Buffer, net.Conn, http.ResponseWriter, ...
// ALL implement io.Writer. Your logger function works with all of them:

func Log(w io.Writer, msg string) {
    fmt.Fprintf(w, "[LOG] %s\n", msg)
}

Log(os.Stdout, "hello")        // prints to terminal
Log(someFile, "hello")       // writes to file
Log(&bytes.Buffer{}, "hi")   // writes to memory buffer

🔬 How Interfaces Work Internally

Internally, an interface value is a pair of two pointers: (type, value).

go 
var p Paymenter          // nil interface: (type=nil, value=nil)

p = Stripe{}            // (type=*Stripe, value=0xc000...)
p = PayPal{}            // (type=*PayPal, value=0xc000...)

// ⚠️ Tricky: a nil concrete pointer inside an interface is NOT nil
var s *Stripe = nil
var pi Paymenter = s   // (type=*Stripe, value=nil)
fmt.Println(pi == nil) // false! The type part is set
⚠️
This "nil interface vs nil pointer in interface" distinction is a common Go gotcha in interviews and production bugs.
Interview Q&A — Interfaces
Q1. How does Go implement interfaces differently from Java or C#? Easy ▼
In Java/C#, a class must explicitly declare that it implements an interface using implements or :. In Go, implementation is implicit — if your type has all the methods the interface requires, it automatically satisfies it, with no declaration needed. This is called structural typing. The benefit is that you can define an interface for a type you don't own, or define the interface after the fact, without touching the original type.
Q2. What is the empty interface and when should you use it? Easy ▼
interface{} (aliased as any in Go 1.18+) specifies zero methods, so every type satisfies it. It's Go's equivalent of Object in Java. Use it when you genuinely don't know or don't care about the type — e.g., generic logging, JSON deserialization into map[string]any, or function parameters that must accept multiple unrelated types. Avoid overusing it — it bypasses compile-time type checking. In Go 1.18+, prefer generics ([T any]) for type-safe containers.
Q3. What is a type assertion and what happens if it fails? Medium ▼
A type assertion extracts the concrete type from an interface: v, ok := i.(T). If the concrete type is T, ok is true and v holds the value. If not, ok is false and v is the zero value of T — no panic. If you use the single-return form v := i.(T) and the assertion fails, Go panics. Always use the two-return form when you're not certain of the type.
Q4. Can an interface be nil in Go? What are the two ways it can be nil? Hard ▼
An interface value is internally a pair (type, value). There are two nil scenarios:
  1. Nil interface: both type and value are nil. A variable declared as an interface but never assigned: var p Paymenter. p == nil is true.
  2. Interface holding a nil pointer: type is set, value is nil. E.g., var s *Stripe; var p Paymenter = s. Now p == nil is false, even though s is nil! Calling a method through p will reach Stripe's method, but the receiver will be nil.
This is a notorious Go gotcha. Return typed nils (return nil, nil) not typed-nil-as-interface to avoid it.
Q5. What design principles do interfaces enable in Go? Medium ▼
Interfaces support several important principles:
  • Open/Closed Principle (OCP) — open for extension (add new types), closed for modification
  • Dependency Inversion Principle (DIP) — high-level code depends on abstractions, not concrete types
  • Liskov Substitution Principle (LSP) — any type satisfying an interface can replace another
  • Testability — inject mocks or stubs implementing the same interface
  • Decoupling — components communicate through contracts, not implementation details
Q6. What is interface composition and give a stdlib example? Medium ▼
Interface composition means embedding one interface inside another to create a new, broader interface. The embedded interface's methods are promoted. Example from the Go standard library:

type ReadWriter interface { Reader; Writer }

Where Reader has Read() and Writer has Write(). ReadWriter requires both. Other examples: io.ReadWriteCloser, io.ReadSeeker. This lets you be precise about what a function actually needs — accept io.Reader not io.ReadWriter if you only read.
Q7. What is the "accept interfaces, return concrete types" principle? Hard ▼
This is idiomatic Go advice coined by Jack Lindamood:
  • Accept interfaces in function parameters — makes your function flexible and testable with any conforming type
  • Return concrete types from functions — gives callers the full API surface without forcing them to type-assert, and avoids leaking abstraction
The exception is when you're designing a library API where you want to hide implementation details or allow multiple return types — then returning an interface is appropriate. But for internal code, returning concrete types is usually better.
